# Blueprint source manifest

This manifest inventories all 32 markdown and Mermaid sources under `blueprint/` and assigns one explicit precedence tier, source role, concern set, and downstream consumer band to each file.

## Precedence summary

| Rank | Tier | Conflict rule |
| --- | --- | --- |
| 1 | `canonical_phase0_foundation` | This tier is the canonical foundation for object shape, invariants, lifecycle control, route intent, release freeze, and assurance trust. |
| 2 | `relevant_phase_blueprint` | Relevant phase files refine workflow behavior for their own phase but must remain compatible with Phase 0 invariants and cross-cutting shell/runtime law. |
| 3 | `specialized_cross_cutting_blueprint` | Cross-cutting blueprints define canonical UI, runtime, admin, support, and specialist operating law whenever a summary layer is broader than the concrete contract. |
| 4 | `programme_phase_cards` | Phase cards align summary language and sequencing but must not override Phase 0, relevant phase files, or specialized blueprints. |
| 5 | `audited_flow_baseline` | The audited flow is a supporting baseline. If it diverges from Phase 0 or a relevant phase blueprint, the canonical blueprint wins and the flow must be corrected. |
| 6 | `forensic_patch_guidance` | Forensic findings must be represented as patch requirements, exclusions, or anti-regression rules; they tighten the corpus but do not authorize weaker semantics. |
| 7 | `orientation_summary` | The bootstrap summary orients later work but never overrides canonical or specialized sources. |
| 8 | `supporting_design_reference` | Supporting design references help interpret the front-end direction but do not override explicit shell, accessibility, runtime, or phase contracts. |
