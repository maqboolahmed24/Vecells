## Prompt
You are a principal systems architect and autonomous remediation agent responsible for fully resolving `Delta-First Evidence Review Presentation`.

Map this domain to `TaskCanvasFrame(summaryStackRef -> deltaStackRef -> evidenceStackRef -> consequenceStackRef -> referenceStackRef)`, `EvidencePrism`, `EvidenceDeltaPacket`, `SelectedAnchor`, `WorkspaceProminenceDecision`, `AttentionBudget`, and any review flow that resumes with changed evidence, changed endpoint, changed lineage, or changed approval posture. Your mission is to fully resolve this failure class. Identify and eliminate every place where reopened review still forces full re-reading, buries consequential deltas, or replaces prior judgment context silently instead of presenting material change first.

Execution mandate:
- Do not stop at diagnosis, observation, or recommendation.
- Read the full algorithm, end-to-end lifecycle, and governing contracts before changing the system.
- Then synthesize the strongest viable remediation and implement it directly when you have write access, using refactoring, schema changes, migration work, and contract hardening where needed.
- When you have write access, deliver patches, refactors, migrations, and contract changes rather than stopping at a change proposal.
- Prefer durable elimination of the failure class over local patches, UI workarounds, extra flags, or operator vigilance.
- Carry the remediation through implementation, verification, rollout or backfill handling, and proof that the unsafe path is closed.

Work method:
- Read the case-review, evidence, prominence, and reopened-review rules before making changes.
- Distinguish quiet review posture, summary-first posture, diff-first posture, decisive or consequential deltas, and contextual or clerical deltas.
- Trace how new evidence lands, how changed-since-seen state is recorded, and how the active canvas chooses which region to promote.
- Inspect how superseded judgment context, prior endpoint assumptions, and selected anchors remain visible when live evidence changes.
- Move from diagnosis into implementation as soon as the governing model is clear; do not leave the task as analysis-only commentary.

Required remediation work:
- Convert the discovered failure modes into a concrete target design and implement that design across the affected layers.
- Remove or refactor the unsafe paths so the system no longer depends on operator vigilance, local convention, or lucky sequencing.
- For each unsafe condition, carry the fix through the owning algorithm, contracts, storage, integrations, and user-facing state until the failure is removed or explicitly governed.
- Find reopened tasks that still default to full history or routine summary instead of diff-first presentation when new material evidence exists.
- Detect `EvidencePrism` or canvas behavior that hides decisive deltas inside long history, collapses change cues too early, or auto-promotes multiple competing support regions at once.
- Surface cases where endpoint, ownership, duplicate-lineage, or approval changes silently replace old context instead of marking supersession explicitly.
- Examine whether delta classes are typed strongly enough to distinguish in-place annotation from commit-invalidating change.
- Identify whether the shell returns to the last quiet posture after conflict or compare resolution, or whether promoted detail lingers and creates review fatigue.

Implementation posture:
- Choose the remediation that definitively closes the failure class, even if that requires non-trivial refactoring.
- Implement the end-state architecture and algorithm, not just the analysis of it.
- Prefer deleting, consolidating, or hardening weak paths over documenting them or layering more conditional logic on top.
- Do not treat delta-first review as a highlighting layer on top of an unchanged reading model.
- Prefer one typed evidence-delta pipeline that makes material change legible before the reviewer revisits full history.
- If sources disagree about what changed, redesign around one authoritative delta packet and one prominence decision contract.
- Ensure old judgment context remains visible with explicit supersession markers until the reviewer rechecks and recommits intentionally.

Produce:
1. A concise description of the current failure mode, its blast radius, and the target end-state.
2. The exact algorithmic, architectural, schema, contract, runtime, and interface changes required to eliminate the issue.
3. The concrete code, migration, configuration, and data-shape changes you will apply, or an exact executable patch plan if the environment is truly read-only.
4. The invariants, state-transition rules, guardrails, and recovery semantics enforced by the new design.
5. The verification suite needed to prove the remediation, including unit, integration, contract, replay, recovery, concurrency, and end-to-end coverage as applicable.
6. The rollout, backfill, reindex, compatibility, and operational validation plan required to land the fix safely.
7. Residual risks only if they remain after implementation, plus the exact evidence required to close them.

Optimize for definitive elimination of `Delta-First Evidence Review Presentation` through coherent algorithmic and architectural remediation, with durable invariants, safe rollout, and proof-backed correctness. The task is not complete at diagnosis; finish only when the strongest viable remediation is implemented and proven, or reduced to an exact executable change set for a truly read-only environment.
